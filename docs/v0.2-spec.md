## 1. AIS PARSER SPEC

### 1.1 Decrypt algorithm (code-ready, Node built-in `crypto`, zero new deps)

Confirmed byte-for-byte across 4 independent OSS impls and proven by a local Node round-trip (peppered password decrypts; un-peppered fails with ERR_OSSL_BAD_DECRYPT).

```
function decryptAis(fileText: string, pan: string, dob: string, explicitPassword?: string): unknown
  k = fileText.trim()
  if k starts with '"': k = JSON.parse(k)            // JSON-string-wrapped payload variant (gist handles this)

  iv   = Buffer.from(k.slice(0, 32),  'hex')          // 16 bytes
  salt = Buffer.from(k.slice(32, 64), 'hex')          // 16 bytes  <-- THIS is the PBKDF2 salt (see R1)
  tail = k.slice(64)

  // ciphertext is base64 OR hex; a hex string also passes the base64 charset test,
  // so build BOTH decodings and try each (canonical impls emit base64)
  ctCandidates = []
  if /^[A-Za-z0-9+/]+=*$/.test(tail): ctCandidates.push(Buffer.from(tail, 'base64'))
  if /^[0-9a-fA-F]+$/.test(tail):     ctCandidates.push(Buffer.from(tail, 'hex'))
  if empty: throw AisFormatError('ciphertext is neither base64 nor hex')

  dob8 = dob.replace(/[-/]/g, '')                     // DDMMYYYY, 8 digits (date of incorporation for non-individuals)
  PEPPER = 'GQ39%*g'                                  // fixed app secret; verified in 4 impls, NOT ITD-documented
  passwords = explicitPassword ? [explicitPassword] : [
    pan.toLowerCase() + PEPPER + dob8,                // primary (all 4 impls)
    pan.toLowerCase() + dob8,                         // fallback: un-peppered (in case live ITD blob differs)
    pan.toUpperCase() + dob8,                         // fallback: uppercase, un-peppered (gist candidate order)
    pan.toUpperCase() + PEPPER + dob8,                // defensive extra (not in any impl, cheap)
  ]

  for pw of passwords:
    key = crypto.pbkdf2Sync(pw, salt, 1000, 32, 'sha256')   // PBKDF2-HMAC-SHA256, 1000 iters, dkLen 32
    for ct of ctCandidates:
      try:
        d = crypto.createDecipheriv('aes-256-cbc', key, iv)  // autoPadding=true strips PKCS#7
        plaintext = Buffer.concat([d.update(ct), d.final()])
        return JSON.parse(plaintext.toString('utf8'))
      catch: continue                                        // bad-decrypt = wrong candidate, keep going
  throw AisDecryptError('all password/encoding candidates failed; supply password explicitly')
```

**Ship gate:** the pepper is verified only against the OSS tools, never against a live ITD blob in this research. Before release, round-trip ONE real AIS JSON locally (Sagar's own, never committed). The un-peppered fallback keeps the tool functional if the live scheme differs.

### 1.2 Decrypted JSON shape (TS interfaces)

The document is a generic label/data grid, NOT fixed field names. Field identity comes ONLY from `columnLabel` strings.

```ts
type AisLabel = string | { name: string };            // BOTH occur; normalize: typeof l === 'string' ? l : l.name

interface AisColumnTable {
  columnLabel: AisLabel[];
  columnData: unknown[][];                            // parallel rows; zip positionally with columnLabel
}

interface AisElement {
  title: string;                                      // information category / code title (free string)
  l1Src?: string;                                     // human-readable source label (deductor / reporting entity)
  l2Src?: string;
  l1?: AisColumnTable;
  l2?: AisColumnTable;
  // fallback: when NEITHER l1 nor l2 present (Part B3 "Payment of Taxes"), columns live on the element:
  columnLabel?: AisLabel[];
  columnData?: unknown[][];
}

interface AisSection { title: string; elements: AisElement[] }

interface AisDocument {
  metadata: Record<string, string>;                   // flat Field/Value pairs
  header: AisColumnTable;
  partA: AisColumnTable;                              // taxpayer info: one data row, transpose to Field/Value
  partB: { sections: AisSection[] };
}
```

**Unresolved research conflict (flagged, do not hard-bake):** one researcher says l1 = summary / l2 = detail; the other says l2 = summary / l1 = transaction rows. Verification did not settle it. Parser must emit BOTH levels generically (`levels: [{level: 'l1', rows}, {level: 'l2', rows}]`) and let the fixture decide which is which; classify empirically on the first real fixture and then document.

**Parse rules:**
- Walk `partB.sections[] -> elements[]`; read `l2` if present, `l1` if present, else element-level columns.
- Zip labels to row values; match fields by label regex (`/amount|value/i`, `/date/i`, `/pan\b/i`, `/tan\b/i`, `/account/i`, `/section/i`, `/status/i`), never by position -- column order is not guaranteed across categories.
- Surface both `Reported value` and `Modified value` columns when present (AIS feedback fields).
- Normalized output per row: `{ sectionTitle, elementTitle, source, level, fields: Record<string,string>, amount?, date?, code? }`.

## 2. FORM 16 PART A SPEC

### 2.1 Chosen PDF lib + API calls

**Primary: `pdf-parse@2.4.5`** (proven locally on a real 4-page TRACES PDF; pdfjs-dist 5.4.296 under the hood; Node >=20.16/22.3).

```ts
import { PDFParse, PasswordException } from 'pdf-parse';
const parser = new PDFParse({ data: buffer, password });   // LoadParameters pass through to pdfjs
const text = await parser.getText();                        // linear text -- use for Part B + part detection
```

- **Do NOT use `getTable()`** -- verified to return ZERO tables on TRACES-format Form 16.
- **Part A needs positioned text** (two-column grid interleaves and splits labels mid-word in linear order: `'PA' + 'RT A'`, `'Amount paid/cred' + 'ited (Rs.)'`). Use the pinned `pdfjs-dist@5.4.296` directly for Part A pages: `getDocument({data, password}).promise` -> `page.getTextContent()` -> items with `str`, `transform[4]`=x, `transform[5]`=y; cluster items by rounded y into rows, sort by x within a row, then run anchors on reassembled rows. (Alternative if this fights: `unpdf@1.6.2` `extractTextItems` exposes x/y and takes `{password}` via `getDocumentProxy` -- undocumented but verified in source.)

### 2.3 Anchors / regex heuristics

Anchors must be SHORT and column-tolerant (never full sentences -- they split across columns):

| Field | Heuristic |
|---|---|
| Part detection | page contains `FORM NO. 16` (tolerate split `FORM`/`NO. 16`) + `PART A` or `PA`+`RT A`; `Certificate under Section 203` as confirmation |
| certificateNo | `/Certificate No\.?\s*:?\s*([0-9A-Z]{6,8})/` on reassembled row |
| lastUpdatedOn | `/Last updated on\s*(\d{2}-[A-Za-z]{3}-\d{4})/` |
| PAN | `/\b([A-Z]{5}[0-9]{4}[A-Z])\b/` -- employee PAN = the match nearest/after an `Employee` label zone; deductor PAN nearest `Deductor` |
| TAN | `/\b([A-Z]{4}[0-9]{5}[A-Z])\b/` |
| AY | `/(?:Assessment\s+Year|A\.Y\.?)[\s:]*(\d{4}-\d{2,4})/` |
| Period | `From`/`To` labels + `/\d{2}-[A-Za-z]{3}-\d{4}/` |
| Quarterly row | reassembled row matching `/^Q[1-4]\b/` with a receipt token (6-15 char alnum, uppercased, must contain a letter, excluding QUARTER/TOTAL/AMOUNT/RECEIPT/NUMBER) and >=3 amounts; `taxDeducted = min(amounts)`, `amountPaidCredited = max(amounts)`; normally assert deducted == deposited |
| Total row | row containing `Total (Rs.)` |
| Challan table | anchored by first-cell `Challan Identification Number (CIN)` / `Book Identification Number (BIN)`; OLTAS legend U/P/F/O |
| Cross-validation | every continuation page repeats `Certificate Number:.. TAN of Employer:.. PAN of Employee:.. Assessment Year:..` -- parse and require consistency (note: real samples contain typos between page-1 and continuation values; mismatch = lower confidence, not hard fail) |

**Part B groundwork (bonus, proven path):** linear `getText()` + items 1-19 anchored by section-number prefix + keyword (`1.`, `2.`, `10(a)`, `13.`), amount capture `([\d,]+\.\d{2})` handling BOTH comma'd and uncomma'd (real sample prints `3107600.00`). Per R4: expect ONE trailing amount per line; attempt two-amount VI-A capture first, fall back to single. Labels can split across lines (`...under section 10\n(10AA) 300000.00`) -- join continuation lines before matching. Regime: `/opting\s+(for|out\s+of)\s+taxation\s+u\/s\s+115BAC(\(1A\))?\??\s*(Yes|No)/i` (FY24-25+ wording is "opting out of ... 115BAC(1A)").

### 2.4 Password try-list

Explicit `password` param always wins. When absent and PDF is encrypted (catch `PasswordException`), and `pan`/`dob` supplied, try in order:

1. `PAN5_UPPER + DDMMYYYY` (primary convention)
2. `pan5_lower + ddmmyyyy`
3. `FULLPAN_UPPER + DDMMYYYY`
4. `fullpan_lower + ddmmyyyy`
5. `last4PANchars + DDMMYYYY` (ClearTax-documented variant)
6. `FULLPAN_UPPER` / `fullpan_lower` alone

Do NOT include employee-ID formats (that claim was unsourced -- verification correction). If all fail, return an actionable error: "Form 16 password is employer-set and varies; check the employer email and pass `password` explicitly."

## 3. RECONCILE SPEC (`reconcile_documents`)

Authority split (verified): **26AS is the ledger of record for TDS credit** (official "rely on TRACES" rule; CPC restricts credit to 26AS); **AIS is the ledger for income lines**; **Form 16 anchors salary**. Notices pre-empted: 143(1)(a) (30-day response; clause (vi) = form income missing from return) and 139(9) (15 days; TDS claimed without income / 26AS receipts > declared).

Tolerances: TDS equality checks exact to the rupee. Income checks: |delta| <= Rs 10 = pass (s.288A/288B rounding slack, the only statutory slack -- no published CPC tolerance exists); Rs 10 < delta < Rs 100 = info; >= Rs 100 = warn; >= Rs 10,000 = high. The warn/high cutoffs are TOOL HEURISTICS, configurable, labeled as such in output.

Whitelist engine (mirrors CBDT Instructions 9/2017, 10/2017, Circular 1/2018): a return figure LOWER than form figures explained by HRA exemption, standard deduction (75K new / 50K old), other s.10 exemptions, or Chapter VI-A deductions is downgraded to "explainable difference", never a finding. Also whitelist: FD interest accrual-vs-receipt, dividend record-date straddle, joint-holder attribution, gross-vs-net professional receipts, exempt income (PPF etc.) in AIS.

## 4. ENGINE UPGRADES

### 4.0 Rounding hierarchy (implement first; everything depends on it)

1. **s.288A**: total income -> nearest Rs 10 (last digit >=5 rounds up).
2. **s.288B**: final tax payable / refund -> nearest Rs 10.
3. **Rule 119A(c)**: interest PRINCIPAL -> truncate DOWN to lower multiple of Rs 100 (`Math.floor(p/100)*100`; the rule says "nearest" but the fraction-ignored clause makes it a floor -- verified via the 3,125 -> 3,100 ITD example).
4. **Rule 119A(b)**: monthly interest, part month = FULL month (always round the period UP).
5. Interest is simple, never compounded; 234A/B/C can all apply simultaneously.

### 4.1 Section 234B

- **Trigger:** advance-tax liability existed (net tax after TDS >= Rs 10,000, s.208) AND (no advance tax paid OR advance tax paid < 90% of assessed tax).
- **Assessed tax** = tax on total income minus TDS/TCS, relief u/s 89/90/90A/91, credits 115JAA/115JD. Advance tax is NOT deducted for the 90% denominator.
- **Interest** = 1%/month (part = full) on `floor100(assessedTax - advanceTaxPaid)`, from 1 April of the AY to the payment/143(1) date; period splits at each self-assessment payment and continues on the reduced balance. Months count Apr=1 ... inclusive of payment month.
- **Exemption:** resident senior citizen (60+) with no PGBP income owes no advance tax -> no 234B/234C ever.

**Golden tests (verbatim):**
- Suraj: tax 28,400, TDS 10,000, assessed 18,400, advance 0, paid 31 Aug of AY -> 5 months -> `18,400 x 1% x 5 = Rs 920`.
- Company nil case: advance-tax base 2,90,000; paid 2,67,000; 90% = 2,61,000; 2,67,000 > 2,61,000 -> **234B = 0**.

### 4.2 Section 234C

- **Base:** tax due on RETURNED income minus TDS/TCS/reliefs/credits. Keep this a separate parameter from 234B's assessed tax even though they coincide pre-assessment.
- **Checkpoints:** cumulative 15% by 15 Jun, 45% by 15 Sep, 75% by 15 Dec, 100% by 15 Mar. Interest = `floor100(shortfall) x 1% x 3 months` for installments 1-3, `x 1 month` for 15 Mar. Fixed 3 months, no proration, no refund on later top-up. Stops at 31 Mar. Not applicable if net liability < Rs 10,000.
- **Safe harbors (statutory, bare-act confirmed):** paid >= 12% of base by 15 Jun -> installment 1 nil; >= 36% by 15 Sep -> installment 2 nil. When BREACHED, shortfall measures from 15%/45% (not 12%/36%). None for Dec/Mar. Write explicit tests at paid = exactly 12%/36% (nil) and 11.99% (interest from 15%).
- **Exceptions (first proviso):** shortfall attributable to capital gains, s.2(24)(ix) winnings (115BB/115BBJ), FIRST-time PGBP income, and dividend per s.2(22) EXCLUDING 2(22)(e) deemed dividend (Finance Act 2021 -- do NOT implement the pre-2021 "115BBDA > 10L" wording some articles still print), provided tax is paid in remaining installments or by 31 Mar. Implement: exclude such income from the base for installments BEFORE the accrual date.
- **Presumptive 44AD/44ADA:** single installment, 100% by 15 Mar, 1-month interest on shortfall.

**Golden tests (verbatim):**
- Khushal (exercises 119A floor): liability 45,500; required 6,825/20,475/34,125/45,500; cumulative paid 8,000/19,000/31,000/45,500; only Dec short: 34,125-31,000 = 3,125 -> floor100 -> 3,100 -> `3,100 x 1% x 3 = Rs 93`. Total 93.
- Company (safe-harbor branch): base 2,90,000; Jun paid 40,000 >= 12% (34,800) nil; Sep 1,05,000 >= 36% (1,04,400) nil; Dec shortfall 12,500 x 3% = 375; Mar shortfall 23,000 x 1% = 230. Total **Rs 605**.
- ClearTax (below-safe-harbor branch): liability 1,00,000; paid 5,000/25,000/35,000/50,000 -> 300 + 600 + 1,200 + 500 = **Rs 2,600** (5% < 12%, shortfall correctly from 15%).
- Scripbox Rs 275: **R6 -- conflicts with the statutory 36% safe harbor** (its Sep charge should be 0 since 40% > 36%). Use per-installment arithmetic only, or adjust expected Sep interest to 0; annotate the test file.

### 4.3 Section 234A (cheap add, formulas researched)

1%/month (part = full) on (tax minus advance tax, TDS/TCS, reliefs, credits, pre-due-date SA tax), from day after the DUE DATE to filing date. Key off the notified due date per AY (CBDT extends mid-season), never a hardcoded Jul 31. Goldens: 8,400 x 5 mo = **Rs 420** (source prints "4 months" but 420 only reconciles at 5 -- flagged discrepancy, 5/420 are the golden values); 23,000 x 1 mo = **Rs 230**.

### 4.4 HRA calculator

**Formula (Rule 2A, verbatim statute):** exempt = least of (a) actual HRA received for the relevant period; (b) rent paid minus 10% of salary; (c) 50% of salary (metro) / 40% (elsewhere). Salary on DUE basis = basic + DA (only if terms of employment provide, i.e. forming part of retirement benefits) + commission at a FIXED percentage of turnover (Gestetner SC). Nothing else.

**Critical rule-pack gotcha:** FY 2025-26 metro list is ONLY **Delhi, Mumbai, Kolkata, Chennai**. The 8-city expansion (adds Bengaluru/Hyderabad/Pune/Ahmedabad) is effective 2026-04-01 = FY 2026-27. Do NOT put 8 cities in `fy2025-26.json` (ClearTax's live page misleads here).

**Period-wise computation:** "relevant period" = months accommodation occupied. Input = array of homogeneous periods `{months, basic, daRetirement, turnoverCommission, hraReceived, rentPaid, isMetro}`; compute least-of-three per period, sum. Annual fast path only when constant for 12 occupied months.

**Regime:** old regime only. `regime: 'new'` -> exemption 0 with explanatory note (HRA fully taxable under 115BAC; 80GG also barred).

**Warnings (not gates):** rent > Rs 1,00,000/yr -> landlord PAN required (Rule 26C/Form 12BB); HRA <= Rs 3,000/month -> employer receipt waiver (CBDT Circulars 8/2013, 20/2015).

**Golden tests (verbatim):** Anwar metro exempt 1,62,000 / same non-metro 1,29,600; Vinod (DA+commission salary base 5,28,000) exempt 91,200; Shiva rent-limb 84,000; Delhi monthly unit 15,000/mo = 1,80,000/yr; Bengaluru-stays-40% regression: exempt 4,80,000 under FY25-26 pack.

**80GG (companion, old regime, no HRA received any time in year):** least of Rs 60,000/yr (5,000/mo), 25% of adjusted total income, rent minus 10% of ATI. ATI = total income before 80GG, excluding LTCG, 111A STCG, 115A/115D income, other VI-A. Output note: Form 10BA mandatory, acknowledgement number in Schedule 80GG. Goldens: 60,000 (cap limb), 50,000 (25% limb), 42,000 (rent limb).
